# 泰语字幕系统架构建议

这份文档是 `docs/thai-asr-eval-loop.md` 的实现侧补充，目标是把“评测闭环”落到系统结构上。

## 推荐分层

### 1. ASR 骨架层

默认使用 Groq：

- 产出 segment
- 产出 word timestamps
- 作为时间轴底座

### 2. 文本修正层

使用 GPT-4o 逐 segment 修正：

- 只修字，不修时间
- 不做整条视频级投影
- 不把错误段污染到别的段

### 3. 泰语增强层

`pythainlp` 仅做增强：

- 断词加空格
- 展示可读性
- 兜底 Romanization

### 4. 翻译层

翻译独立：

- 原文字幕先稳定
- 中文翻译后生成
- 不反向影响原文

## 推荐数据流

```text
视频/音频
  → 提取音轨 / 切片
  → Groq 生成 segments + words
  → segment 级文本修正（GPT-4o，逐段裁决）
  → 泰语加空格 / 词级展示增强
  → 翻译
  → 导出字幕 / 播放
```

## 不建议保留的模式

- 整条视频一次性投影到 Groq 骨架
- 失败后继续沿用旧播放对象
- 把 `pythainlp` 当成主识别依赖
- 在前端暴露底层模型名和对齐细节

## 设计目标

- 泰语字幕更接近标准答案
- 时间轴稳定
- 失败可见
- 用户体验优先

## 当前部署结论

- 默认主链路继续使用 Groq，作为全球默认方案。
- Qwen3-ASR-Flash-Filetrans 已在项目内打通，但只建议作为中国内地环境下的高精度候选。
- 海外环境下不把 Qwen 作为默认主链路，避免区域依赖和额外延迟影响用户体验。
- 等用户回到中国内地测试环境后，再按同一套评测标准比较 Groq 与 Qwen。
