# 泰语 ASR 评测闭环与新系统设计

> 适用范围：泰语识别、分句、分词、翻译、字幕导出与播放体验。
>
> 核心原则：只看最终字幕文件是否接近标准答案，不看内部模型名是否“高级”。

## 1. 为什么要有标准集

泰语链路的错误通常不是全局性的，而是局部段落、局部对齐、局部翻译污染。
如果没有标准答案，就只能靠主观感觉判断，最后会不断打补丁，却不知道是否真的变好。

因此需要一个固定的 ground truth 数据集：

- 视频文件
- 同名或可配对的标准 `.srt` 字幕

评测的目标不是“某个模型更强”，而是：

- 哪条链路生成的最终字幕最接近标准答案
- 哪种错误最常出现
- 哪一层应该修，而不是继续堆补丁

## 2. 评测对象

每次改动后，至少比较三条链路：

1. `Groq-only`
   - 只用 Groq 生成最终字幕
   - 作为时间轴与段落稳定性的基线

2. `OpenAI Whisper-only`
   - 旧 OpenAI Whisper 路线
   - 作为纯 OpenAI 时间戳路线的对照组

3. `current_mix`
   - GPT-4o Transcribe 文本修正 + Groq 断句骨架
   - 这是当前最值得继续优化的候选方案

## 3. 评分指标

评测时不只看“识别有没有成功”，还要看最终字幕和标准答案的距离。

建议固定四类指标：

- 文本相似度：最终字幕文本 vs 标准字幕文本
- 误差率：字符级编辑距离
- 断句质量：segment 数量与标准字幕 cue 数量是否接近
- 失败率：下载失败、识别失败、翻译失败是否被正确拦截

对于泰语，还要额外观察：

- 是否漏词
- 是否把泰语识别成夹杂中文/英文的碎片
- 是否出现过短 segment 或异常长 segment
- 是否出现“前一步失败，后一步继续拿旧结果播放”的错误

## 4. 评测闭环

推荐把整个过程固定成下面的循环：

1. 选择标准集
2. 跑三条链路生成最终字幕文件
3. 与标准 `.srt` 比对
4. 输出每条样本的差异和汇总分数
5. 定位错误类型
6. 只改最该改的那一层
7. 再跑同一标准集

这会形成真正的正向循环：

标准字幕 → 自动评测 → 找问题段 → 改规则 → 再评测 → 收敛

## 5. 当前的阶段性结论

在已经跑过的泰语样本里：

- `current_mix` 多数情况下最接近标准字幕
- `Groq-only` 通常稳定，但文本质量较粗
- `OpenAI Whisper-only` 在泰语样本上通常更弱，也更慢

这意味着当前的优化方向不是“回到 Whisper-only”，而是：

- 保留 Groq 的时间轴底座
- 保留 GPT-4o 的文本修正收益
- 去掉脆弱的整条投影式拼接

## 6. 新系统设计

### 6.1 识别层

默认由 Groq 产生：

- `segments`
- `words`

角色：

- 提供稳定时间轴
- 提供可用的词级时间戳
- 作为整个字幕系统的骨架

### 6.2 文本修正层

GPT-4o 只做逐 segment 修正，不做整条视频的全局投影。

原则：

- 一段一段地修
- 不改时间轴
- 不把某一段的结果错误扩散到其它段

### 6.3 泰语后处理层

`pythainlp` 只作为增强工具：

- 泰语按词加空格
- 泰语词级展示
- 罗马拼音兜底

它不是主识别引擎，也不应该成为系统的单点依赖。

### 6.4 翻译层

翻译必须独立于识别：

- 原文识别成功后再翻译
- 翻译失败不能污染原文
- 翻译失败不能回放旧字幕或旧视频

## 7. 逐 segment 裁决规则

每个 segment 都要单独决定：

- 保留原识别文本
- 使用修正后的文本
- 只重识别这个 segment

建议接受修正文本的条件：

- 更完整
- 更像泰语
- 不明显缩水
- 不混入异常语言

不满足条件时，宁可保留原 segment，也不要把坏文本扩散到整条视频。

## 8. 失败隔离

任何步骤失败都必须停在当前状态，不能静默继续：

- 下载失败：停止
- 识别失败：停止
- 翻译失败：停止
- 导出失败：停止

禁止的行为：

- 用旧视频顶替新视频
- 用旧字幕顶替新字幕
- 失败后继续播放“看起来像成功”的结果

## 9. 对前端的要求

前端只展示用户能理解的状态：

- 正在识别
- 正在校正
- 正在翻译
- 完成
- 失败，请重试

不向普通用户展示：

- Groq / Whisper / GPT-4o 名称
- 投影、对齐、tokenization 等实现细节

这些信息对用户没有直接价值，只会增加等待成本和理解成本。
