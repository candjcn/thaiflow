踩过的坑

坑 #1 — #2

坑 #1
AI 会"记忆混乱"
跨越多次对话后，AI 引用了一个早已被删除的函数 _gemini_request，导致线上直接报错崩溃。
✓ 解法：用 CLAUDE.md 给 AI 一份"持久记忆"，每次对话开头它都会读
坑 #2
功能越加越脆
加了一个新按钮，发现旧的播放逻辑莫名其妙坏了。AI 改了它不应该改的地方，而且不主动告诉你。
✓ 解法：让 AI 先读懂现有代码再动手，明确要求"保持向后兼容"

踩过的坑

坑 #3 — #4

坑 #3
AI 爱"过度设计"
让 AI 加一个简单按钮，它给你加了 3 层抽象、工厂模式、配置项……代码量翻 5 倍，反而更难维护。
✓ 解法：在文档里明确写："三行重复代码优于一个不必要的抽象"
坑 #4
UI 越改越乱
AI 改了一个按钮位置，把另一个功能的交互逻辑也默默改了。文字说"完成了"，但手机上根本不对。
✓ 解法：每次 UI 变更后必须在真实设备上测，不能只听 AI 汇报

踩过的坑 · 最隐蔽的一个

坑 #5：第三方 API 版本陷阱

现象：yt-dlp 版本太旧，YouTube 悄悄升级了反爬机制，下载功能整体失效，用户收到一堆英文报错。


更糟的是：AI 写的代码能跑，但它不会主动追踪外部依赖的版本变化。第三方坏了，你要自己发现。


✓ 解法：把版本固定依赖改为"每次部署拉最新版"；同时把英文报错翻译成用户能看懂的中文提示。

AI 写的代码不会自我维护 — 这是你作为"产品负责人"要承担的责任

送给 AI 初学者

4 条真实建议

01
先做一个自己真正想用的工具，不要做 Demo。真实需求才能驱动真实迭代。
02
写清楚你的项目说明文档（CLAUDE.md）。AI 的"长期记忆"靠的是它，而不是你的期待。
03
每次 UI 变更后，在真实设备上验证。AI 说"完成了"不等于用户体验没有倒退。
04
不要害怕出错。248 次提交里有 87 次在修 Bug — 出错是进步的证据，不是失败的信号。
